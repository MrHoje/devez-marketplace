<Purpose>
Deep-Ask는 Ouroboros 방식의 소크라테스식 질문과 수학적 모호성 점수 측정을 구현한다. 타겟 질문으로 숨은 가정을 드러내고, 가중치 차원별로 명확도를 측정하며, 모호성이 임계값 이하로 떨어질 때까지 진행을 거부한다. 출력은 게이트 파이프라인으로 전달된다: **deep-ask → hoje-plan 합의 정제 → 승인 대기 → 명시적 실행 승인**.
</Purpose>

<Execution_Policy>
- **1회 1질문** 원칙 — 절대 여러 질문을 배치로 묶지 않음
- 질문 전 항상 **가장 취약한 명확도 차원**을 타겟팅
- Round 1 전에 **Round 0 토폴로지 확인 게이트**를 1회 실행
- 매 라운드 취약 차원을 명시: 차원명, 점수/갭, 해당 질문 이유
- 코드베이스 사실은 사용자에게 묻기 전 **먼저 직접 탐색**
- brownfield 확인 질문 시 사용자가 재발견하게 하지 말고 증거 인용
- 답변 후 **모호성 점수를 투명하게 표시**
- 멀티 컴포넌트일 경우 각 컴포넌트별 점수 측정, 순회하며 타겟팅
- 프롬프트 예산 관리: 질문/점수/스펙 작성 전 과도한 컨텍스트는 요약
- 모호성 ≤ 임계값 + 사용자 명시적 승인 전까지 실행 금지
- "구현", "implementation plan" 등은 실행 허락이 아닌 최종 목표 기술로 간주
- 인터뷰 중 코드 작성/수정/worker 실행 금지, 계속 범위/리스크/수용 기준 질문
- 조기 종료 허용 (경고 후)
- 인터뷰 상태를 세션 중단에도 복구 가능하게 유지
- 마일스톤 전환 시 **다중 페르소나 횡단 검토 패널** 가동
- 자유 텍스트 답변은 구조화하여 해석, 누락 확인 후 점수 측정
- **자동 답변 3회 연속**: 다음 질문은 반드시 사용자에게 직접
- 스펙 결정화 전 **독립 완결 확인** + **한 문장 목표 재확인**, 각각 사용자 확인 필요
</Execution_Policy>

## Phase 0: 모호성 임계값 확인

기본값 `0.05` (5%) 사용. 첫 줄에 반드시 출력:

```
Deep-Ask 임계값: 5% (source: default)
```

## Phase 0.5: 적합성 게이트

요청이 이미 구체적이고, 파일 경로/함수명/수용 기준이 명시된 단일 변경이면:
1. 즉시 종료 — 상태 초기화하지 말고 직접 실행 경로로 반환
2. "요청이 명확하고 작은 작업이므로 deep-ask가 필요하지 않습니다" 안내
3. 사용자가 계속 요청하면 Phase 1 진행

## Phase 1: 초기화

1. 아이디어 파싱, greenfield/brownfield 판단
2. brownfield면: 코드베이스 탐색, 기존 deep-ask/hoje-plan 산출물 확인
3. 과도한 컨텍스트는 요약, 원본은 외부 참조로 보관
4. 상태 초기화 후 인터뷰 시작 안내

## Round 0: 토폴로지 확인

점수 측정 전 1회 실행. 최상위 컴포넌트(1-6개) 추출하여 확인:

```
Round 0 | 토폴로지 확인 | 점수: 아직 미측정

{N}개 최상위 컴포넌트로 읽었습니다:
1. {이름}: {설명}
2. ...

이 구성이 맞나요? 추가/제거/병합/분리할 컴포넌트가 있나요?
```

확정 후 각 컴포넌트별로 상태 저장.

## Phase 2: 인터뷰 루프

모호성 ≤ 임계값까지 반복:

### 2a. 질문 생성
- 활성 컴포넌트 중 가장 낮은 점수의 차원 타겟
- N > 1개 컴포넌트가 비슷하면 순회
- **가정(assumption)**을 노출하는 질문

### 2b. 질문
```
Round {n} | 컴포넌트: {name} | 타겟: {dim} | 이유: {rationale} | 모호성: {score}%

{question}
```

### 2c. 모호성 점수 측정
Greenfield: `모호성 = 1 - (목표×0.40 + 제약×0.30 + 기준×0.30)`
Brownfield: `모호성 = 1 - (목표×0.35 + 제약×0.25 + 기준×0.25 + 맥락×0.15)`

**양방향·비단조**: 새 답변이 기존 확정 사실과 모순되면 점수 상승 가능.

### 2d. 진행 보고
라운드별 차원 점수 표 + 다음 타겟 안내.

## Phase 3: 횡단 검토 패널 (마일스톤 전환 시)

| 밴드 | 모호성 범위 |
|------|------------|
| initial | > 60% |
| progress | 60% ≥ a > 30% |
| refined | 30% ≥ a > 임계값 |
| ready | ≤ 임계값 |

전환 시 4개 페르소나(researcher/contrarian/simplifier/architect)를 병렬 소환하여 맹점 탐지.

## Phase 4: 스펙 결정화

### 4a. 완결 확인
- 모든 컴포넌트 커버리지 확인
- 미해결 트리거 점검
- 갭 발견 시 Phase 2로 복귀

### 4b. 목표 재확인
"이 한 줄이 당신이 원하는 것과 동일한가요?"

### 4c. 스펙 파일 생성
`.hoje/specs/deep-ask-{slug}.md`에 저장. 포함 항목: 메타데이터, 명확도 분석, 토폴로지, 목표, 제약, 비목표, 수용 기준, 가정/결론, 인터뷰 기록.

## Phase 5: 실행 연결

```
스펙 완성 (모호성: {N}%). 어떻게 진행할까요?
```

**옵션:**
1. **hoje-plan으로 계획 정제 (권장)**
2. **hoje-goals로 직접 실행**
3. **계속 인터뷰**

명시적 선택 없이 직접 구현 금지. 조기 종료 조건: Round 3+ 사용자 요청 시 경고 후 종료, Round 10 경고, Round 20 강제 종료.
