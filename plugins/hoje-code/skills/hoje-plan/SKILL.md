# Gate-Plan (Consensus Planning)

> **우선순위 규칙**: 이 스킬이 로드되면 `writing-plans` 스킬을 자동 호출하지 않는다.
> 계획 수립은 gate-plan이 담당한다. writing-plans는 사용자가 명시적으로 요청할 때만 허용.

실행 요청을 받으면 먼저 **구체성 게이트**를 통과시킨다. 모호하면 합의 계획을 수립하고, 구체적이면 즉시 통과한다.

## 게이트 판단: 통과 vs 차단

**통과 (즉시 실행 가능)** — 아래 중 하나라도 있으면:
- 파일 경로 포함 (`src/hooks/bridge.ts`)
- 이슈/PR 번호 (`#42`)
- 함수명, 클래스명, 변수명
- 번호 매긴 단계 목록
- 수용 기준(acceptance criteria) 명시
- 에러 메시지 포함
- `force:` 또는 `!` 접두사

**차단 → 계획 수립 필요:**
- "이거 고쳐줘", "앱 만들어줘", "성능 개선해줘", "인증 추가해줘"
- 파일명, 함수명, 기준 없이 15단어 이내

## 계획 수립 워크플로우

### 준비
요청을 받으면 코드베이스를 탐색해 관련 파일, 패턴, 제약사항을 파악한다.

### Step 1: Planner — 초안 작성
Agent로 Planner를 스폰하여 다음을 포함한 계획 초안을 작성한다:
- **원칙** (3-5개): 이 계획이 따르는 설계 원칙
- **결정 기준** (상위 3개): 선택의 우선순위 기준
- **실행 가능한 옵션** (2개 이상): 각 옵션의 장단점
- **권장 접근**: 어떤 옵션을 왜 선택하는지
- **수용 기준**: 완료를 판단할 수 있는 구체적 기준 (테스트 가능해야 함)

### Step 2: Architect 검토
Step 1 완료 후 (병렬 금지), Agent로 Architect를 스폰하여:
- 아키텍처 건전성 검토
- 가장 강력한 반론(steelman antithesis) 제시
- 실제 트레이드오프 긴장 최소 1개
- 원칙 위반 여부 명시
- 결과: `CLEAR` / `WATCH` / `BLOCK`

### Step 3: Critic 평가
Step 2 완료 후 (병렬 금지), Agent로 Critic을 스폰하여:
- 원칙-옵션 일관성 검토
- 리스크 완화 명확성
- 수용 기준의 테스트 가능성
- 결과: `APPROVE` / `ITERATE` / `REJECT`

### 재검토 루프 (최대 5회)
Critic이 `ITERATE` 또는 `REJECT`이면:
1. Architect + Critic 피드백 수집
2. Planner에게 피드백 전달하여 계획 수정
3. Architect 재검토 → Critic 재평가
4. `APPROVE`가 나오거나 5회 초과 시 종료

5회 초과 시 최선의 버전을 사용자에게 제시.

## 최종 계획 출력

Critic이 `APPROVE`하면 계획을 `pending approval` 상태로 마크하고 사용자에게 제시:

```markdown
## 계획 요약
[한 문장 목표]

## 접근 방법
[선택된 옵션과 근거]

## 실행 단계
1. ...
2. ...

## 수용 기준
- [ ] ...

## 리스크 & 트레이드오프
- ...

상태: PENDING APPROVAL
```

**중요:** 계획은 `pending approval` 상태. 사용자가 명시적으로 승인해야 실행.

## 승인 후 실행

사용자 승인 시:
- 기본: `run-goals` 스킬을 Skill 툴로 호출하여 계획 기반 실행
- 직접 구현하지 말 것

## 사용법

```
/hoje-plan "인증 시스템 추가"           ← 모호 → 계획 수립
/hoje-plan "fix src/auth/jwt.ts:42"    ← 구체 → 즉시 통과
/hoje-plan --interactive "..."         ← 각 단계에서 사용자 확인
```

## 트러블슈팅

| 문제 | 해결 |
|------|------|
| 구체적인데 차단됨 | 파일 경로나 함수명 추가 |
| 우회하고 싶음 | `force:` 접두사 추가 |
| 계획이 너무 추상적 | Architect에게 구체적 파일명 요구 |
