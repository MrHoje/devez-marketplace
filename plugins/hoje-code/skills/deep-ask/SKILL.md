# Deep-Ask (Ambiguity Interview)

> **우선순위 규칙**: 이 스킬이 로드되면 `brainstorming` 스킬을 자동 호출하지 않는다.
> 요구사항 명확화는 deep-ask가 담당한다. brainstorming은 사용자가 명시적으로 요청할 때만 허용.

아이디어를 받아 **모호성 점수**가 임계값 이하로 내려갈 때까지 1회 1개 질문으로 인터뷰한다.

## Phase 0: 임계값 설정 (필수 첫 단계)

기본값 `0.05` (5%) 사용.

**첫 줄 필수 출력:**
```
Deep-Ask 임계값: 5%
```

## Phase 1: 초기화

1. 아이디어 파싱
2. Explore agent로 **greenfield** (새 구축) vs **brownfield** (기존 코드 수정) 판단
3. brownfield면: 관련 코드베이스 영역 탐색하여 컨텍스트 수집
4. 인터뷰 시작 안내 출력:

```
Deep-Ask 임계값: 5%

요청을 완전히 이해하기 위해 집중 질문을 드립니다.
각 답변 후 명확도 점수를 보여드립니다.
모호성이 5% 이하로 내려가면 스펙을 작성합니다.

아이디어: "{initial_idea}"
프로젝트 유형: {greenfield|brownfield}
현재 모호성: 100%
```

## Round 0: 토폴로지 확인 게이트

Phase 2 점수 측정 전, 딱 한 번 실행.

아이디어에서 최상위 컴포넌트(1-6개)를 추출하여 확인 요청:

```
Round 0 | 토폴로지 확인 | 점수: 아직 미측정

이 아이디어를 {N}개 최상위 컴포넌트로 읽었습니다:
1. {이름}: {한 줄 설명}
2. ...

이 구성이 맞나요? 추가/제거/병합/분리/보류할 컴포넌트가 있나요?
```

AskUserQuestion 또는 직접 질문으로 확인 받고 컴포넌트 목록 확정.

## Phase 2: 인터뷰 루프 (모호성 ≤ 임계값까지 반복)

### 2a. 다음 질문 생성

**타겟팅 전략:**
- 4개 차원 중 가장 낮은 점수의 컴포넌트+차원을 타겟
- 여러 컴포넌트가 비슷하게 낮으면 컴포넌트를 교대(rotate)
- 기능 목록이 아닌 **가정(assumption)**을 노출하는 질문

**차원별 질문 스타일:**
| 차원 | 스타일 |
|------|-------|
| 목표 명확성 | "~할 때 사용자가 처음 하는 구체적 행동은?" |
| 제약 명확성 | "오프라인 동작이 필요한가, 인터넷 연결을 가정하는가?" |
| 수용 기준 | "무엇이 되면 '됐다'고 말할 수 있나?" |
| 맥락 (brownfield) | "기존 src/auth/의 패턴을 확장하나, 새로 만드나?" |

**Greenfield + research 필요 질문:** `deep-ask-greenfield` 스킬을 Skill 툴로 호출하여 후보를 얻어 질문 옵션으로 제시.

**연속 3회 자동 답변 후:** 다음 질문은 반드시 사용자에게 직접 묻기.

### 2b. 질문하기

```
Round {n} | 컴포넌트: {이름} | 타겟: {차원} | 이유: {한 줄} | 모호성: {N}%

{질문}
```

**사용자가 "알아서 해줘" / 답변 위임 시:**
`deep-ask-auto-answer` 스킬을 Skill 툴로 호출하여 단일 결정 획득 후 적용.
단, 이 결정으로 임계값을 넘을 것 같으면 사용자에게 명시적 확인 요청.

**자유 텍스트 답변 시 (복잡한 내용):**
구조화하여 확인:
- **결정**: ...
- **근거**: ...
- **제약 (사용자 명시)**: ...
- **범위 외 (사용자 명시)**: ...

한 번 확인 후 진행.

### 2c. 모호성 점수 계산

**Greenfield 공식:**
```
모호성 = 1 - (목표 × 0.40 + 제약 × 0.30 + 기준 × 0.30)
```

**Brownfield 공식:**
```
모호성 = 1 - (목표 × 0.35 + 제약 × 0.25 + 기준 × 0.25 + 맥락 × 0.15)
```

각 차원은 0.0~1.0. 답변이 기존 확인된 사실과 **모순**되면 해당 차원 점수를 낮춰 모호성 상승.

**점수 상승 트리거:**
- A. 기존 확인 사실과 직접 모순
- B. 두 요구사항이 공존 불가
- C. 회피적/모호한 답변
- D. 범위 확장 (새 컴포넌트/통합 추가)

### 2d. 진행 보고

```
Round {n} 완료.

| 차원 | 점수 | 가중치 | 가중 합 | 갭 |
|------|------|--------|---------|-----|
| 목표 | {s} | 0.40 | {s*0.4} | {갭} |
| 제약 | {s} | 0.30 | {s*0.3} | {갭} |
| 기준 | {s} | 0.30 | {s*0.3} | {갭} |
| **모호성** | | | **{이전}% → {현재}%** | |

다음 타겟: {컴포넌트} / {차원} — {이유 한 줄}

{점수 ≤ 임계값 ? "임계값 도달! 스펙 작성 준비됨." : "다음 질문 집중 영역: {차원}"}
```

## Phase 3: 횡단 검토 패널 (마일스톤 전환 시)

**마일스톤 밴드:**
| 밴드 | 모호성 범위 |
|------|------------|
| `initial` | > 60% |
| `progress` | 60% ≥ a > 30% |
| `refined` | 30% ≥ a > 임계값 |
| `ready` | ≤ 임계값 |

마일스톤 전환 시, `deep-ask-panel` 스킬을 Skill 툴로 호출하여 4개 페르소나 소환.
패널 결과 중 구체적 발견사항을 다음 단일 질문 옵션에 통합.

**모호성 3라운드 정체 시:** 패널에게 온톨로지 재프레이밍 지시 ("핵심 엔티티가 무엇인가?").

## Phase 4: 스펙 결정화

**모호성 ≤ 임계값 도달 시:**

### 4a. 완결 확인 게이트
스펙 생성 전 독립적으로 검토:
- 모든 활성 컴포넌트에 목표/제약/기준 커버리지 있는가?
- 미해결 트리거가 있는가?
- 낮은 신뢰도 자동 답변이 사실로 오인되지 않는가?

갭 발견 시: "수학적으로는 준비됐지만 {갭}이 있어 아직 수용 불가" 선언 후 Phase 2로 복귀.

### 4b. 목표 재확인
인터뷰 전체를 한 문장으로 압축:
"이 한 줄만 보면 당신이 원하는 것과 동일한 결과에 도달할 수 있나요?"

### 4c. 스펙 파일 생성

저장 경로: `docs/deep-ask-{slug}.md`
`docs/` 폴더가 없으면 자동 생성 후 저장.

```markdown
# Deep-Ask 스펙: {제목}

- 최종 모호성: {N}%
- 유형: greenfield|brownfield
- 임계값: 5%

## 명확도 분석
| 차원 | 점수 | 가중치 | 가중 합 |

## 토폴로지
| 컴포넌트 | 상태 | 설명 | 커버리지 |
```

## Phase 5: 실행 연결

스펙 저장 후 AskUserQuestion으로 다음 단계 선택:

```
스펙 완성 (모호성: {N}%). 어떻게 진행할까요?
```

**옵션:**
1. **gate-plan으로 계획 정제 (권장)** — Planner/Architect/Critic 합의 후 실행 승인 대기
2. **run-goals로 직접 실행** — 스펙이 이미 구체적이고 단순한 경우만
3. **계속 인터뷰** — 더 명확히 하고 싶을 때

선택 후:
- gate-plan 선택 → `gate-plan` 스킬을 Skill 툴로 호출 (스펙 파일 컨텍스트 전달)
- run-goals 선택 → `run-goals` 스킬을 Skill 툴로 호출
- **명시적 선택 없이 직접 구현 금지**

## 조기 종료 조건

- **Round 3+**: 사용자가 "됐어", "시작해", "만들어줘" → 경고 후 종료 허용
- **Round 10**: 경고 표시 후 계속/진행 선택
- **Round 20**: 강제 종료 후 현재 모호성으로 스펙 생성
